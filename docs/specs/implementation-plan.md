# Implementation Plan

- **Project:** Cortex
- **Version:** 0.1
- **Date:** 2026-03-03

## Overview

Cortex is a self-hosted, agent-readable second brain. Users capture thoughts from Telegram (text or voice) or a web editor. Every entry is classified by Claude, embedded by Ollama, and stored in PostgreSQL with pgvector. Any AI tool can read, write, and search the brain via MCP.

The implementation follows a "capture first" strategy: get Telegram ingestion working end-to-end before building the web interface or MCP server. Each feature goes through the full spec-dd workflow (behavioral spec, test spec, test implementation spec, failing tests, then implementation) before moving to the next.

The project is greenfield. Scale is small (< 1,000 entries). There is no existing code.

## Implementation Phases

### Phase 1: Foundation

**Features:** Configuration, Logging, Database, Health Endpoint

**Work:**

- Set up project scaffolding: `package.json`, `tsconfig.json`, `Dockerfile`, `docker-compose.yml` (PostgreSQL with pgvector, Ollama, faster-whisper), `.env.example`
- Config module (`src/config.ts`): load env vars, validate required values, apply defaults, expose typed config object
- Logger (`src/logger.ts`): structured JSON logging with timestamp, level, module, message, and context fields
- Database (`src/db/`): Drizzle schema for `entries` and `settings` tables, migrations, connection setup with pgvector extension
- Health endpoint (`GET /health`): returns status of PostgreSQL, Ollama, whisper, and Telegram components with uptime
- Hono server entrypoint (`src/index.ts`): start server, wire up health route

**Dependencies:** None

**Spec-dd features:** `foundation`

---

### Phase 2: Intelligence

**Features:** Embedding, Classification

**Work:**

- Ollama integration (`src/embed.ts`): connect to Ollama, generate embeddings with `qwen3-embedding` (4096 dimensions), connectivity check on startup, automatic model pull if not present (`ollama pull qwen3-embedding`)
- Claude API classification (`src/classify.ts`): send thought text with classification prompt, parse structured JSON response, context-aware classification using last 5 recent entries and top 3 semantically similar entries
- Embedding retry logic: entries stored with `embedding: null` are retried, exponential backoff on Ollama failures

**Dependencies:** Phase 1 (database for storing/querying entries, config for API keys and URLs)

**Spec-dd features:** `embedding`, `classification`

---

### Phase 3: Telegram Capture

**Features:** Telegram Bot (text, voice, fix command, inline category buttons)

**Work:**

- grammY bot setup (`src/bot.ts`): long-polling mode, no public URL required
- Chat ID validation: check sender against authorized list from settings table / `TELEGRAM_CHAT_ID` env var
- Text message flow: receive message, classify with Claude (with context), embed with Ollama, store in PostgreSQL, reply with confirmation
- Voice message flow: download OGG/Opus audio, transcribe with faster-whisper (`src/transcribe.ts`), then same pipeline as text with `source_type: 'voice'`
- Confidence-based replies: high confidence (>= threshold) shows checkmark with category and name; low confidence shows question mark with inline keyboard of 5 category buttons
- Inline category correction: user taps button, entry category is updated, fields re-generated by Claude, embedding re-generated
- `/fix` command: find most recent entry, re-classify with correction context from user's message, update entry
- Error handling: database down returns "System temporarily unavailable", transcription failure returns "Could not transcribe voice message. Please send as text."

**Dependencies:** Phase 2 (classification and embedding), Phase 1 (database and config)

**Spec-dd features:** `telegram-bot`

---

### Phase 4: Web Application

**Features:** Auth, Dashboard, Browse, Entry, New Note, Trash, Settings, SSE

**Work:**

- Hono web server routes (`src/web/routes.ts`): serve all pages and API endpoints
- Tailwind CSS CLI build (`tailwind.config.ts`, `src/web/public/style.css`): pre-built CSS, no runtime overhead
- Auth middleware (`src/web/auth.ts`): login page, session-based cookies using `SESSION_SECRET`, `WEBAPP_PASSWORD` env var, all routes except `/login` and `/health` require authentication
- Dashboard page (`/`): today's digest (always visible), recent entries (last 7 days grouped by day), quick stats (entries this week, open tasks, stalled projects), quick capture input with markdown support, live updates via SSE
- Browse page (`/browse`): entries by category, semantic search bar (cosine similarity >= 0.5), text search fallback (name/content matching), tag filtering with autocomplete
- Entry page (`/entry/:id`): view with rendered markdown, edit mode (textarea, category/tags/fields), soft delete with confirmation
- New note page (`/new`): full markdown editor, manual or AI-suggested category, tag autocomplete from existing tags, "AI Suggest" button for classification
- Trash page (`/trash`): list soft-deleted entries, restore button (sets `deleted_at` to null), empty trash button (hard delete)
- Settings page (`/settings`): authorized Telegram chat IDs, classification model, digest schedule (daily/weekly cron), timezone, confidence threshold, digest email recipient, Ollama URL
- SSE endpoint (`src/web/sse.ts`): push events for new entries, updates, deletes, and digest refreshes

**Dependencies:** Phase 2 (classification and embedding for quick capture and AI suggest), Phase 1 (database for all data operations)

**Spec-dd features:** `web-auth`, `web-dashboard`, `web-browse`, `web-entry`, `web-new-note`, `web-settings`

---

### Phase 5: MCP Server

**Features:** MCP stdio + HTTP transport, 7 tools

**Work:**

- Stdio entrypoint (`src/mcp.ts`): MCP server communicating over stdin/stdout for local AI tool integration (e.g., `docker exec -i cortex-app-1 node dist/mcp.js`)
- 7 MCP tools:
  - `search_brain`: semantic search across all entries (cosine similarity >= 0.5, configurable limit)
  - `add_thought`: capture a thought with full classification pipeline (context-aware, source: 'mcp')
  - `list_recent`: browse entries from last N days, optional category filter
  - `get_entry`: read a specific entry by UUID
  - `update_entry`: update name, content, category, tags, or fields
  - `delete_entry`: soft-delete an entry
  - `brain_stats`: entry counts by category, recent activity, open tasks, stalled projects
- Streamable HTTP transport at `/mcp`: for remote access, uses same session authentication as webapp
- HTTP auth: same cookie-based auth as web application

**Dependencies:** Phase 2 (classification and embedding for `add_thought` and `search_brain`), Phase 1 (database for all tools)

**Spec-dd features:** `mcp-server`

---

### Phase 6: Digests & Automation

**Features:** Daily digest, Weekly review, Email, Cron

**Work:**

- Daily digest generation (`src/digest.ts`): query database for active projects with next actions, people with pending follow-ups, tasks due within 7 days, entries captured yesterday; send to Claude with daily digest prompt; format as plain text (150 words max, sections: TOP 3 TODAY, STUCK ON, SMALL WIN)
- Weekly review generation: query past 7 days of entries and activity stats; send to Claude with weekly review prompt; format as plain text (250 words max, sections: WHAT HAPPENED, OPEN LOOPS, NEXT WEEK, RECURRING THEME)
- Email sending (`src/email.ts`): nodemailer SMTP wrapper, send digest to configured recipient; on failure log error, do not retry (next scheduled digest will have current data)
- Cron scheduling (`src/cron.ts`): configurable via env vars and settings table; daily digest default `30 7 * * *`, weekly review default `0 16 * * 0`
- Embedding retry cron: every 15 minutes, process entries with `embedding: null`
- Classification retry cron: process entries with `category: null`
- Dashboard digest caching: store latest digest for display on dashboard, push via SSE on generation

**Dependencies:** Phase 2 (classification for retry cron and digest generation), Phase 1 (database and config)

**Spec-dd features:** `digests`

## Spec-DD Workflow Per Feature

For each feature listed in the spec-dd feature map, follow this process in order:

1. **Write behavioral specification** (Phase 1 of spec-dd) -- Define what the feature does from the user's perspective. Capture every behavior, edge case, and error scenario. Output: `docs/specs/{feature}-specification.md`
2. **Derive test specification** (Phase 2 of spec-dd) -- Translate each behavior from the specification into a concrete test case with description, given/when/then, and expected outcome. Output: `docs/specs/{feature}-test-specification.md`
3. **Write test implementation specification** (Phase 3 of spec-dd) -- Specify exactly how each test will be implemented: file paths, helper functions, mocks, assertions, setup/teardown. Output: `docs/specs/{feature}-test-implementation-specification.md`
4. **Implement tests** -- Write the actual test code following the test implementation specification. All tests must FAIL initially (red phase).
5. **Implement feature** -- Write the production code to make all tests pass (green phase).
6. **Review alignment** -- Verify the implementation matches the behavioral specification. Check for gaps, update specs if requirements evolved.

No feature implementation begins until its tests exist and fail. No tests are written until the three specification documents are complete.

## Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| qwen3-embedding not available on Ollama | Blocks all embedding and semantic search | Fall back to snowflake-arctic-embed2 (1024 dims) or nomic-embed-text:v1.5 (768 dims); adjust the `vector(4096)` column to match; make model configurable via `OLLAMA_MODEL` env var |
| faster-whisper container resource usage | Slow transcription on low-spec hardware, high memory consumption | Make voice capture optional, allow disabling whisper in config, document minimum hardware requirements |
| Claude API rate limits | Slow classification during burst capture, blocked digest generation | Batch classification requests, implement retry with exponential backoff, queue entries for async processing |
| SSE connection management | Stale connections consuming server resources, missed events | Implement heartbeat pings every 30 seconds, client-side reconnection logic, event ID tracking for replay |
| Ollama cold start | First embedding request slow after container restart, model loading delay | Pull model during startup sequence, health check waits for model availability before marking ready |
| pgvector HNSW index on empty table | No issue (HNSW supports empty tables unlike IVFFlat) | Already mitigated by architecture choice |

## Docker Development Setup

The development environment uses Docker Compose for infrastructure services and runs the application directly on the host for fast iteration.

**Start infrastructure:**

```bash
docker-compose up -d postgres ollama whisper
```

This starts:
- **PostgreSQL** (pgvector/pgvector:pg16) on port 5432 with health checks
- **Ollama** (ollama/ollama) on port 11434 for local embeddings
- **faster-whisper** (fedirz/faster-whisper-server) on port 8000 for voice transcription

**Pull the embedding model (first time only):**

```bash
docker exec cortex-ollama-1 ollama pull qwen3-embedding
```

**Run the application:**

```bash
npm run dev
```

Starts the app with hot reload. Connects to PostgreSQL, Ollama, and whisper running in Docker. Runs Drizzle migrations automatically on startup.

**Build Tailwind CSS (watch mode):**

```bash
npm run tailwind:watch
```

Rebuilds `src/web/public/style.css` on file changes.

**Run the full stack in Docker (production-like):**

```bash
docker-compose up
```

Builds the app image and starts all four services together. The app waits for PostgreSQL to be healthy before starting.

**Run tests:**

```bash
npm test
```

Tests use a separate test database and mock external services (Ollama, Claude API, faster-whisper, Telegram API).
