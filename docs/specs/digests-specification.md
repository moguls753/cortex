# Digests - Behavioral Specification

| Field | Value |
|-------|-------|
| Feature | Digests |
| Phase | 6 |

## Objective

Generate AI-powered daily briefings and weekly reviews. Deliver via email and dashboard. Automate background maintenance (embedding and classification retries).

The digest system runs three scheduled jobs via node-cron: a daily morning briefing, a weekly review, and a background retry loop for failed embeddings and classifications. The daily digest surfaces the most important items from the brain (active projects, pending follow-ups, upcoming tasks, yesterday's captures) and delivers a concise, actionable summary. The weekly review summarizes the past 7 days of activity and highlights open loops. Both digests are generated by Claude, cached for dashboard display, pushed to connected browsers via SSE, and sent via email over SMTP. The background retry job ensures that entries which failed embedding or classification (due to temporary Ollama or Claude outages) are eventually processed without manual intervention.

## User Stories & Acceptance Criteria

### US-1: Daily Morning Briefing

**As a user, I want a daily morning briefing that surfaces my most important items.**

- **AC-1.1:** A cron job triggers at the configured daily schedule. The default cron expression is `30 7 * * *` (07:30 daily). The expression is read from the `digest_daily_cron` key in the settings table, falling back to the `DAILY_DIGEST_CRON` environment variable, falling back to the hardcoded default. The cron job runs in the timezone configured by the `timezone` setting (default `Europe/Berlin`).
- **AC-1.2:** The system reads the following data from the database:
  - Active projects with next actions (`category: 'projects'`, `fields->>'status': 'active'`, where `fields->>'next_action'` is not null or empty).
  - People with pending follow-ups (`category: 'people'`, where `fields->>'follow_ups'` is not null or empty).
  - Tasks due within 7 days (`category: 'tasks'`, `fields->>'status': 'pending'`, where `fields->>'due_date'` is within the next 7 days).
  - Entries captured yesterday (any category, `created_at` within the previous calendar day).
  - All queries exclude soft-deleted entries.
- **AC-1.3:** This data is formatted and sent to Claude (using the model configured by the `anthropic_model` setting, default `claude-sonnet-4-20250514`) with the daily digest prompt. The prompt instructs Claude to produce a plain text briefing with a maximum of 150 words and 3 fixed sections: TOP 3 TODAY, STUCK ON, SMALL WIN.
- **AC-1.4:** Claude returns a plain text digest. The digest is validated to be non-empty. The format contains sections as specified by the daily digest prompt: `TOP 3 TODAY`, `STUCK ON`, and `SMALL WIN`. The response is used as-is (no post-processing of Claude's output beyond trimming whitespace).
- **AC-1.5:** The digest is cached for dashboard display. The cache stores the latest daily digest content and its generation timestamp. Only the most recent daily digest is kept; previous ones are overwritten.
- **AC-1.6:** The digest is pushed to all connected dashboard clients via Server-Sent Events (SSE). Connected browsers update the digest section without requiring a page refresh.
- **AC-1.7:** The digest is sent via email to the configured recipient (see US-3).

### US-2: Weekly Review

**As a user, I want a weekly review summarizing my past week.**

- **AC-2.1:** A cron job triggers at the configured weekly schedule. The default cron expression is `0 16 * * 0` (Sunday 16:00). The expression is read from the `digest_weekly_cron` key in the settings table, falling back to the `WEEKLY_DIGEST_CRON` environment variable, falling back to the hardcoded default. The cron job runs in the configured timezone.
- **AC-2.2:** The system reads the following data from the database:
  - All entries created in the past 7 days (any category, excluding soft-deleted).
  - Activity statistics: count of entries per day for the past 7 days, count of entries per category for the past 7 days.
  - Stalled projects: `category: 'projects'`, `fields->>'status': 'active'`, `updated_at` older than 5 days. Excludes soft-deleted entries.
- **AC-2.3:** This data is formatted and sent to Claude with the weekly review prompt. The prompt instructs Claude to produce a plain text review with a maximum of 250 words and 4 fixed sections: WHAT HAPPENED, OPEN LOOPS, NEXT WEEK, RECURRING THEME.
- **AC-2.4:** Claude returns a plain text review. The review is validated to be non-empty. The response is used as-is (no post-processing beyond trimming whitespace).
- **AC-2.5:** The review is cached for dashboard display. The cache stores the latest weekly review content and its generation timestamp. Only the most recent weekly review is kept; previous ones are overwritten. The weekly review is stored separately from the daily digest (both can be displayed on the dashboard simultaneously).
- **AC-2.6:** The review is sent via email to the configured recipient (see US-3).

### US-3: Email Delivery via SMTP

**As a user, I want digest emails delivered via SMTP.**

- **AC-3.1:** The system sends email using nodemailer with SMTP settings configured via environment variables: `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`. The sender address is `DIGEST_EMAIL_FROM` (defaults to `SMTP_USER`). The recipient is `DIGEST_EMAIL_TO` (overridable via the `digest_email_to` setting in the settings table).
- **AC-3.2:** The email subject for a daily digest is: `"Cortex Daily — {date}"` where `{date}` is the current date formatted as `YYYY-MM-DD` (e.g., `"Cortex Daily — 2026-03-03"`).
- **AC-3.3:** The email subject for a weekly review is: `"Cortex Weekly — w/c {date}"` where `{date}` is the Monday of the current week formatted as `YYYY-MM-DD` (e.g., `"Cortex Weekly — w/c 2026-02-23"`).
- **AC-3.4:** The email body is the plain text digest or review content. No HTML formatting, no headers, no footers. The plain text content from Claude is used directly.
- **AC-3.5:** If SMTP sending fails (connection error, authentication failure, recipient rejected), the error is logged with structured JSON logging (timestamp, level, module: `email`, message, error details). The digest is still cached on the dashboard and pushed via SSE. Email failure does not affect dashboard delivery.
- **AC-3.6:** Failed emails are NOT retried. The next scheduled digest will include current data, effectively making the missed email obsolete.

### US-4: Background Retry for Failed Embeddings and Classifications

**As a system, I want background retry for failed embeddings and classifications.**

- **AC-4.1:** A cron job runs every 15 minutes (cron expression: `*/15 * * * *`). This schedule is not user-configurable.
- **AC-4.2:** The job finds all active (non-deleted) entries where `embedding IS NULL` and generates embeddings for them using Ollama (snowflake-arctic-embed2). The embedding is computed from the entry's `content` field (or `name` if content is empty).
- **AC-4.3:** The job finds all active (non-deleted) entries where `category IS NULL` and re-classifies them using Claude with the classification prompt. The context-aware pipeline is used: last 5 recent entries and top 3 similar entries are fetched as context (similar entries may not be available if the entry itself has no embedding, in which case only recent entries are used).
- **AC-4.4:** Successfully processed entries are updated in the database immediately. Embedding and classification updates are independent: an entry can succeed at embedding but fail at classification (or vice versa) in the same cycle.
- **AC-4.5:** Failed entries (Ollama timeout, Claude error, malformed response) are logged with structured JSON logging and left for the next cycle. No entry is marked as permanently failed.
- **AC-4.6:** Each cycle processes a maximum of 50 entries total (across both embedding retries and classification retries) to avoid overloading Ollama or Claude. If there are more than 50 entries needing processing, the remaining entries are deferred to the next cycle. Entries are processed in `created_at` order (oldest first).

### US-5: Configurable Digest Schedules

**As a user, I want digest schedules to be configurable.**

- **AC-5.1:** Daily and weekly cron expressions are configurable via environment variables (`DAILY_DIGEST_CRON`, `WEEKLY_DIGEST_CRON`). These serve as defaults.
- **AC-5.2:** The settings page in the webapp can override these schedules by writing to the `digest_daily_cron` and `digest_weekly_cron` keys in the settings table. Settings table values take precedence over environment variables.
- **AC-5.3:** When schedules change via the settings page, the running cron jobs are rescheduled without requiring an application restart. The old cron job is stopped and a new one is started with the updated expression.
- **AC-5.4:** Default schedules: daily at 07:30 (`30 7 * * *`), weekly Sunday 16:00 (`0 16 * * 0`). These defaults apply when neither the settings table nor environment variables provide a value.

## Constraints

### External Dependencies

- Digests use the Claude API, which is the only external paid dependency. Both daily and weekly digest generation make one Claude API call each per scheduled run.
- Email delivery requires valid SMTP configuration. If SMTP environment variables are not set, email is skipped entirely (see edge cases).
- Embedding retry depends on Ollama being available. Classification retry depends on Claude API being available.

### Scheduling

- Cron expressions use the standard 5-field format: `minute hour day-of-month month day-of-week`. The node-cron library is used for parsing and scheduling.
- The timezone from the `timezone` setting (or `TZ` environment variable, default `Europe/Berlin`) affects when cron jobs fire. All cron expressions are interpreted in this timezone.

### Data Format

- Digest content is plain text only. No markdown, no HTML. Claude is instructed to produce plain text output.
- The daily digest has a 150-word maximum. The weekly review has a 250-word maximum. These limits are enforced via the prompt, not programmatically. Claude is trusted to respect the word limit.
- Digest sections are fixed per the architecture document. Daily: TOP 3 TODAY, STUCK ON, SMALL WIN. Weekly: WHAT HAPPENED, OPEN LOOPS, NEXT WEEK, RECURRING THEME.

### Performance

- The background retry job processes a maximum of 50 entries per cycle. This limit prevents overloading Ollama (which runs locally and may be resource-constrained) and avoids excessive Claude API usage.
- Digest generation involves one database query phase (multiple queries) and one Claude API call. The total time depends on Claude's response latency (typically 2-10 seconds).

## Edge Cases

### Daily Digest

- **First day with no entries (empty database):** The digest prompt still runs. The data payload to Claude indicates zero items in every category. Claude responds with a message like `"No activity yet — capture your first thought today"` (as guided by the prompt's instruction to be honest about no recent activity).
- **All projects are stalled (active but not updated in > 5 days):** The STUCK ON section lists all stalled projects. Claude is given the full list and formats it according to the prompt rules.
- **No tasks due soon:** The TOP 3 TODAY section focuses on projects with next actions and people with pending follow-ups instead of due tasks.
- **No stuck items:** Claude responds with `"Nothing stalled — good momentum"` in the STUCK ON section (as specified in the daily digest prompt).

### Weekly Review

- **Week with zero entries captured:** Claude responds with something like `"Quiet week — no entries captured"` (as guided by the prompt's instruction to acknowledge quiet weeks without judgment).
- **Very busy week (> 100 entries):** All entries are included in the data payload to Claude. Claude summarizes according to the 250-word limit.

### Email

- **SMTP not configured (all `SMTP_*` environment variables are empty or unset):** Email sending is skipped entirely. No error is logged (this is a valid configuration for users who do not want email). The digest is cached on the dashboard and pushed via SSE only.
- **SMTP configured but recipient (`DIGEST_EMAIL_TO`) is empty:** Email sending is skipped. A warning is logged.
- **SMTP connection timeout:** The error is logged. The digest is still cached on the dashboard.
- **SMTP authentication failure:** The error is logged. The digest is still cached on the dashboard.

### Claude API Failures

- **Claude API is down during daily digest generation:** The error is logged. An error message is cached on the dashboard (e.g., `"Digest generation failed — will retry at next scheduled time"`). No email is sent (there is no content to send). The system does not retry immediately; the next scheduled digest will include current data.
- **Claude API is down during weekly review generation:** Same behavior as daily digest failure.
- **Claude API returns malformed output (not valid text, empty response):** Treated the same as Claude being down. Error is logged, error message cached, no email sent.

### Background Retry

- **Embedding retry finds more than 50 entries with null embeddings:** Only the 50 oldest entries (by `created_at`) are processed in this cycle. The remaining entries are deferred to the next cycle (15 minutes later).
- **Classification retry finds entries but Claude is down:** All classification attempts in this cycle fail. Entries are left with `category: null` for the next cycle. Errors are logged.
- **Ollama is down during embedding retry:** All embedding attempts in this cycle fail. Entries are left with `embedding: null` for the next cycle. Errors are logged.
- **An entry has both null embedding and null category:** Both are attempted in the same cycle (embedding first, then classification). If embedding succeeds but classification fails, the entry is partially updated (embedding is saved, category remains null).
- **No entries need retry:** The cron job runs, finds zero entries to process, and completes immediately with no database writes and no API calls.

### Scheduling

- **Cron expression is invalid (set via settings page):** The invalid expression is rejected at the settings page level. The previous valid schedule remains active. An error message is shown to the user on the settings page.
- **Application starts in the middle of the day (after the daily digest time has passed):** The daily digest is NOT retroactively generated. The system waits for the next scheduled time. For example, if the app starts at 10:00 and the daily digest is scheduled for 07:30, the first digest runs at 07:30 the next day.
- **Application restarts while a digest is being generated:** The in-progress digest is lost. The next scheduled run will generate a fresh digest. No duplicate detection is needed.
- **Timezone changes via settings:** The cron jobs are rescheduled with the new timezone. The next firing time is recalculated based on the new timezone.
- **Two scheduled jobs fire at the same time (e.g., daily and weekly both at 16:00 on Sunday):** They run independently. Each makes its own database queries and Claude API call. No mutex or locking is needed (single-user system).

### SSE

- **No dashboard clients are connected when a digest is generated:** The digest is cached normally. When a client connects, it loads the cached digest on page load. The SSE push is a no-op if no clients are connected.
- **Dashboard client disconnects during SSE push:** The SSE manager handles disconnection gracefully. The digest is already cached, so the client will see it on next page load.

## Non-Goals

- **Digest customization.** The sections for daily (TOP 3 TODAY, STUCK ON, SMALL WIN) and weekly (WHAT HAPPENED, OPEN LOOPS, NEXT WEEK, RECURRING THEME) reviews are fixed as specified in the architecture document. Users cannot add, remove, or rename sections.
- **HTML email formatting.** Emails are plain text only. No HTML templates, no inline styles, no images.
- **Digest history or archive.** Only the latest daily digest and the latest weekly review are cached. There is no database table for historical digests. Previous digests are overwritten.
- **Push notifications beyond email.** No mobile push notifications, no Slack integration, no webhook notifications. Email and dashboard are the only delivery channels.
- **Manual digest trigger from Telegram.** There is no Telegram command to generate a digest on demand. Manual triggers are only possible from the webapp if that feature is implemented in the dashboard.
- **Embedding retry backoff.** The retry interval is a fixed 15 minutes. There is no exponential backoff, no per-entry retry counter, no maximum retry limit. This is sufficient for a single-user system with fewer than 1,000 entries.
- **Digest delivery confirmation.** There is no mechanism to verify that the user read the email or viewed the digest on the dashboard.
- **Configurable word limits.** The 150-word (daily) and 250-word (weekly) limits are fixed in the prompts and not user-configurable.
- **Multi-recipient email.** Digests are sent to a single configured recipient. There is no support for multiple email addresses or distribution lists.
